# FireWorldBench v1 开发约束总则

版本：`1.0-draft`  
生效日期：`2026-07-14`  
规范词：`MUST`=必须，`MUST NOT`=禁止，`SHOULD`=原则上应，偏离需记录理由。

## 1. 范围与事实源

- MUST 只实现当前 v1 的 T1/T2/T3 与已批准配套工程。
- MUST 以 T1/T2/T3 作为统一能力 ontology；MUST NOT 将数据集身份直接定义为任务身份。
- MUST 将仿真、实验、传感器、视觉和多模态视为证据来源/验证角色，并通过统一 Schema 映射到能力任务。
- MUST 保持 v1 为火灾领域 benchmark；跨其他物理领域的扩展留给后续版本。
- MUST NOT 读取或引入 `4.升级拓展`，除非用户在当前或后续对话明确解除限制。
- MUST 把核心 PDF 作为概念基线，把 `详细设计规划`、版本化配置和 Schema 作为执行规格。
- MUST 在需求冲突时停止相关实现、登记冲突并请求决策；MUST NOT 静默自行融合。
- MUST NOT 将本 benchmark 用作真实消防系统决策依据。
- MUST 先运行 benchmark 基线并完成冻结失败分析，再决定论文方法方向；候选方法不得在基线证据前被写成既定贡献。

## 2. 数据治理

- MUST 将 `3.数据集` 视为只读上游；原始文件不得覆盖、重命名或就地修复。
- MUST 为每个进入构建流程的数据源登记来源 URL、获取日期、版本/提交、许可证、文件哈希和本地状态。
- MUST NOT 将海报、README 图、论文截图、示例页或无法验证来源的演示素材作为训练/验证/测试样本。
- MUST NOT 使用许可证未查明、禁止再分发或伦理条件不满足的数据生成公开 benchmark。
- MUST 保留原始层、标准化层、样本层、发布层的不可逆边界；每个派生物必须记录父哈希和转换版本。
- MUST 统一单位并保留原单位；任何单位猜测均视为数据错误，不得自动填补。
- MUST 把空标签、缺失值、传感器失效、截断和无火负样本区分编码，不得互相代替。
- SHOULD 通过内容哈希和 manifest 管理数据版本；不得依赖文件修改时间作为版本依据。

## 3. ID、匿名化与防泄漏

- MUST 在任何切窗、增强、采样或 QA 派生前，按原始 case/scenario 分组完成 split。
- MUST 保证同源重复图、同一视频相邻帧、同一仿真 case 的窗口、同一反事实 family 不跨越禁止的 split 边界。
- MUST 使用与原始文件名无语义关联的公开 `sample_id/case_id`；私有映射与公开包分离。
- MUST 在训练和验证阶段屏蔽冻结测试标签、答案、physical trace、评分元数据和私有映射。
- MUST 运行精确哈希、感知哈希、时间邻近、case family 和问题模板泄漏检查，并保存报告。
- MUST NOT 在查看测试结果后重新选择主指标、阈值、样本过滤规则或 OOD 轴；必要变更须重建版本并废弃旧测试结论。

## 4. 样本与标签

- MUST 使每个样本包含 `scenario`、`observations`、`question`、`answer`、`physical_trace`、`scoring_metadata` 和 `provenance`。
- MUST 区分直接观测、规则派生、模拟真值、文献知识和专家标注；置信度不能伪装成真值。
- MUST 为反事实 pair 验证“只改变一个已声明干预变量”；其他条件变化则不得标为单变量反事实。
- MUST 让答案、物理链、证据引用和自动评分字段可独立校验。
- MUST 对开放式物理解释执行双人校准、协议一致性统计与争议裁决；领域专家人数不足时不得宣称专家级 ground truth。
- SHOULD 优先构建可自动评分的选择、排序、方向和结构化 JSON 任务；开放文本只作为受控补充。
- MUST NOT 把数据集特有的简单 QA 当作独立能力；任何 QA 都必须映射到统一能力、gold origin、证据和评分定义。

## 5. 任务与评测

- MUST 保持 T1、T2、T3 的任务边界，不得以通用 fire/smoke detection 结果替代物理理解结果。
- MUST 在主实验前冻结任务定义、主/次指标、阈值、统计检验和失败处理规则。
- MUST 报告分任务、分来源、ID/OOD、难度和模态切片；MUST NOT 只给单一总分掩盖能力差异。
- MUST 报告不确定性，默认以 scenario/case 为重采样单位计算 95% bootstrap 置信区间。
- MUST 保存每个样本的原始模型输出、解析状态、工具调用、延迟、token/费用和评分明细。
- MUST 将无效 JSON、拒答、超时、工具失败和内容过滤按预注册规则计分，不得人工挑选成功响应。
- MUST 对 API/非确定模型运行预注册的重复次数并报告方差；模型名、快照日期、参数和系统提示必须冻结。
- MUST NOT 用隐藏测试集反复调 prompt、工具或阈值。

## 6. Baseline 公平性

- MUST 包含 chance/majority 或领域阈值下限、任务专用传统/深度模型、纯 LLM 以及至少一个增强 LLM/agent 设置。
- MUST 保证输入信息预算可比较；不同设置使用额外信息或工具时必须明确标为不同 track。
- MUST 报告模型可访问的数据、外部检索语料、工具和训练方式；闭源模型不得使用模糊别名。
- MUST 对 prompt、解析器、重试和后处理版本化；禁止只报告最佳随机种子或最佳 prompt。
- SHOULD 同时报能力、成本、延迟和失败率，避免只按准确率排名。

## 7. 代码与配置

- MUST 使用 Python `>=3.11`，UTF-8，类型注解和 `pathlib`；禁止硬编码个人绝对路径。
- MUST 通过配置表达数据根、split、任务、指标和随机种子；代码默认值不得覆盖冻结配置。
- MUST 使用稳定排序、显式随机种子和内容寻址；同一输入/版本应生成相同 manifest。
- MUST 将核心模块放在 `src/fireworldbench`，测试放在 `tests`，一次性分析也应形成可追踪脚本。
- MUST 通过 Ruff；核心构建、split、Schema、metrics 模块必须通过 mypy。
- MUST 对 split、匿名化、pair 构造、指标和防泄漏逻辑编写正常/边界/失败测试。
- SHOULD 保持总体行覆盖率不低于 80%，上述高风险模块不低于 90%；覆盖率不能替代语义测试。
- MUST NOT 捕获异常后静默继续；错误信息需包含阶段、稳定 ID 和恢复建议，但不能泄露私有标签。

## 8. 环境、运行与复现

- MUST 记录代码提交、脏工作区状态、Python/依赖锁、OS、硬件、随机种子、配置哈希、数据 manifest 哈希和模型版本。
- MUST 为每次正式 run 建立不可覆盖目录，至少包含 `run_manifest.json`、配置快照、原始预测、指标、日志和校验和。
- MUST 从 clean Git commit 执行正式 benchmark/pilot/main run；UNBORN 或 dirty 源码只能用于预研。经 freeze 决策批准的不可变源码归档是唯一例外，必须记录全量 manifest 与 SHA-256。
- MUST NOT 手工编辑正式 run 的指标文件；修复评分器后必须以新 run/version 重算。
- MUST 在论文冻结前进行一次干净环境从 manifest 到表格的端到端复现。
- 密钥 MUST 只通过环境变量或本地秘密存储提供；日志、配置和 Git 中禁止出现密钥。

## 9. 论文与报告

- MUST 让每项主张映射到固定实验、表/图、代码、配置和统计结果；映射登记在 `项目治理/论文证据矩阵.md`。
- MUST 区分探索性与验证性实验，探索结果不得事后伪装为预注册假设。
- MUST 报告数据限制、domain gap、负结果、失败案例、人工标注局限和潜在安全误用。
- MUST 验证所有引用、数据规模和许可证原文；未经核验的信息只能标记为待确认。
- MUST 在双盲阶段移除作者身份、本地路径、私有仓库、运行用户名和可反向识别元数据。

## 10. AI/跨对话协作

- MUST 按 `AGENTS.md` 开始和结束每轮工作。
- MUST 基于实际文件和命令输出汇报；不得凭记忆声称测试、下载或实验已完成。
- MUST 保护用户未提交改动，不得执行破坏性回滚。
- MUST 在每轮结束更新当前状态、会话记录和下一轮提示词；未满足时任务状态只能是 `IN_PROGRESS` 或 `BLOCKED`。
- MUST 按共同 handoff ID 的原子顺序写会话记录、下一提示词、当前状态；三者不一致时新窗口只能进入恢复模式。
- MUST 把阻塞条件写成可解除的具体请求，而不是“后续再看”。
