# 00 总体架构与实现设计

## 1. 设计目标

FireWorldBench v1 需要同时满足四个目标：

1. 科学有效：测量火灾物理状态、机制与演化，不退化为表面检测或记忆问答。
2. 防泄漏：数据、问题、答案、反事实 family 和测试评分在生命周期内隔离。
3. 可复现：从上游文件到论文表格的每一步都有版本、哈希和机器检查。
4. 可扩展：v1 的三任务共享一个样本契约，但数据源、模态和模型适配器可插拔。

## 2. 两级交付边界

### 2.1 `v1-mvp`

用途是验证闭环，不作为最终论文强结论的充分条件。

- 主数据：本地 Immersed-Tunnel-CFD 子集。
- 真实约束：PolyUFire sidewall vent 与少量 FDS-exp。
- 视觉辅助：D-Fire 的合规抽样。
- T1：3/6/10/20 秒窗口的预警、归因和信息选择原型。
- T2：火源区域、排烟配置/主导机理和物理一致性原型。
- T3：同一火源不同排烟配置的方向性 pair 原型。

### 2.2 `v1-paper`

满足投稿主实验所需的完整版本：

- FDS + FD-Gen 形成受控、可复现、单变量干预的 T3 主体和 OOD 场景。
- 完整许可与数据快照，独立冻结测试集。
- 领域专家校准 physical trace 和开放式解释评分。
- 传统、深度、LLM、多模态和 tool-use baseline 全部在统一 harness 下运行。
- 完成 ID/OOD、反事实、消融、稳定性、成本和误差分析。

若 `v1-paper` 门禁无法通过，必须收缩论文主张，而不是用 MVP 数量放大替代。

## 3. 系统分层

```text
只读上游数据
  -> inventory / license gate / hashing
  -> raw manifest (L0, 不复制或只受控缓存)
  -> standardized records (L1, 单位/坐标/字段规范)
  -> case graph + derived events (L2, 物理事件与干预关系)
  -> benchmark examples (L3, T1/T2/T3 JSONL)
  -> split/release package (L4, public inputs + private labels)
  -> model adapters / tool sandbox / inference harness
  -> parser / validator / scorer
  -> statistics / error analysis / paper tables
```

任何层只允许从前一层和版本化配置生成；禁止从模型输出反向修改 gold 数据。

## 4. 逻辑组件

| 组件 | Python 包 | 职责 | 关键不变量 |
|---|---|---|---|
| Inventory | `fireworldbench.data.inventory` | 扫描、哈希、来源/许可状态 | 原始文件只读；稳定相对路径 |
| Adapters | `fireworldbench.data.adapters` | 解析 Immersed、PolyU、FDS、视觉源 | 单位显式；缺失不猜测 |
| Normalizer | `fireworldbench.data.normalize` | 统一时间、单位、坐标、传感器表 | 原值和变换均可追溯 |
| Case graph | `fireworldbench.data.case_graph` | 组织 base case、干预、family、事件 | 单变量 pair 可验证 |
| Splitter | `fireworldbench.splits` | group split、OOD holdout、泄漏扫描 | split 先于窗口/增强 |
| Builders | `fireworldbench.tasks.{t1,t2,t3}` | 生成任务样本、trace、评分字段 | Schema 合法、确定性构建 |
| Validators | `fireworldbench.validation` | Schema、语义、泄漏、物理不变量 | 失败即阻断发布 |
| Adapters | `fireworldbench.models` | 统一传统模型、LLM、多模态 API | 输入预算和版本透明 |
| Tools | `fireworldbench.tools` | 绘图、检索、公式、受控 FDS 代理 | 白名单、超时、完整轨迹 |
| Harness | `fireworldbench.runner` | 运行、重试、缓存、成本和原始输出 | 不覆盖 run；失败可计分 |
| Scoring | `fireworldbench.scoring` | 解析、逐样本指标、聚合和 CI | 评分器版本冻结 |
| Reporting | `fireworldbench.reporting` | 表格、图、误差分析、证据映射 | 只读正式 run 产物 |

当前仓库只建立最小骨架；组件随 P1-P4 逐步实现，不提前制造空 API。

## 5. 目录设计

```text
v1/
  configs/                 # 项目、数据资格、split、evaluation 配置
  schemas/                 # sample/prediction JSON Schema
  src/fireworldbench/      # 可复用实现
  tests/                   # 单元、属性、契约和小型集成测试
  scripts/                 # 薄入口和项目审计，不放核心逻辑
  data/                    # 说明/manifest；raw 默认忽略
  artifacts/               # 本地构建和 run，默认忽略
  详细设计规划/            # 可执行设计
  开发要求约束/            # 硬约束与完成定义
  进度跟进记录/            # 当前状态与追加式会话证据
  项目治理/                # 决策、风险、问题、资产与论文证据
```

## 6. 配置与版本

- 配置使用 TOML；正式 run 保存解析后的完整快照，不依赖隐式默认值。
- 版本由以下五元组决定：`code_commit + data_manifest_hash + benchmark_version + scoring_version + run_config_hash`。
- benchmark 版本与代码包版本分离；数据/任务不兼容变化提升 benchmark 主版本。
- 所有列表在哈希前稳定排序，浮点序列化规则固定，时间使用 UTC ISO 8601。

## 7. 计划 CLI

```text
fwb doctor                       环境、配置和外部依赖检查
fwb data inventory              生成只读数据 manifest
fwb data validate               字段、单位、缺失和许可证门禁
fwb split build                 生成 group/OOD split
fwb benchmark build             构建某个冻结 benchmark 版本
fwb benchmark validate          Schema、泄漏、pair、物理规则检查
fwb run                         统一 baseline/LLM/tool-use 运行
fwb score                       从原始预测生成逐样本和聚合指标
fwb report                      生成冻结表格、图和证据索引
```

每条写命令默认写到新目录；只有显式 `--overwrite-nonfrozen` 可覆盖未冻结临时产物，正式 run 永不覆盖。

## 8. 模型运行协议

### 8.1 Track

- `closed-book`：只给当前样本公开观测。
- `retrieval`：允许访问已冻结、无测试答案的知识库。
- `plot`：允许将公开传感器表转换为图，不增加信息。
- `tool-use`：允许白名单公式/绘图/查询/FDS 代理；记录每次调用。
- `oracle-ablation`：仅研究诊断，可给 gold 子信息，不进入主排行榜。

不同 track 不共用排名；工具错误、超时和额外信息预算必须披露。

### 8.2 Run 产物

每个 run 目录至少包含：

```text
run_manifest.json
resolved_config.toml
environment.json
input_manifest.json
raw_responses.jsonl
predictions.jsonl
sample_scores.jsonl
metrics.json
logs/
checksums.sha256
```

## 9. 错误与恢复

- 数据/Schema/泄漏错误：fail closed，不生成发布包。
- 单模型 API 错误：按预注册重试；耗尽后保留失败记录并计分。
- 解析错误：保存原文和解析器版本，不人工修答。
- 评分器 bug：旧结果标记 invalid，新版本全量重算。
- FDS 失败：保存输入、日志和退出码；不得只保留成功 case 形成选择偏差。

## 10. 非功能要求

- 可移植：路径来自配置，Windows/Linux 均可运行。
- 可审计：公开输入和私有评分资产分区，所有变换可追溯。
- 可测试：高风险规则先有不变量测试，再实现批量构建。
- 可扩展：新增来源实现 adapter，不修改通用样本语义。
- 成本可控：试运行先用固定小集；正式预算在 run 前冻结。
- 双盲安全：发布检查扫描作者名、用户名、绝对路径、私有 URL 和元数据。
