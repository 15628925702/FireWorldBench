# 01 数据与样本协议

## 1. 数据源角色与当前资格

以下是初始化判断，不替代 P1 的许可证和逐文件 manifest。

| ID | 来源 | 当前本地状态 | v1 角色 | 当前计分资格 |
|---|---|---|---|---|
| D01 | Immersed-Tunnel-CFD | 8 个 CFD CSV 抽样 | T1/T2 主体，T3 配置 pair 原型 | `candidate` |
| D02 | PolyUFire | 仓库/实验表已获取 | 真实温度与机理外部验证 | `candidate` |
| D03 | FDS-exp | 5 个实验 CSV 子集 | 物理一致性 sanity/external validation | `candidate` |
| D04 | FD-Gen | 工具仓库已获取，尚非生成数据 | paper-ready 受控场景/T3 | `generator_candidate` |
| D05 | D-Fire | 小型图像/YOLO 抽样 | T1/T2 视觉辅助和干扰 | `candidate_aux_only` |
| D06 | DetectiumFire | real_fire/metadata 抽样，缺 non-fire/video | 多模态 grounding 辅助 | `blocked_incomplete` |
| D07 | Fire360 | 只有官方 poster | 未来视觉时序候选 | `ineligible_placeholder` |
| D08 | MS-FSDB | 只有官方统计图 | 未来检测候选 | `ineligible_placeholder` |
| D09 | DFS | 只有官方示例/清单 | 未来干扰检测候选 | `ineligible_placeholder` |

`candidate` 只表示技术上存在样本；许可、版本、单位和质量门禁通过后才可变为 `eligible`。

### 1.1 异构来源的统一角色

- 仿真数据提供可控参数、完整场/隐状态和候选 counterfactual truth，主要支撑机制、演化和单变量干预；仿真真值不得被表述为真实世界测量。
- 受控实验数据提供真实噪声、缺失、传感器误差和 simulator-to-reality external validation；没有完整状态或干预对时，不强行生成 T3 真值。
- 传感器时序提供弱信号、响应顺序和趋势证据，主要支撑 T1/T2，并在 horizon 和标签可验证时支撑 T3。
- 视觉数据提供火焰/烟气外观、位置、遮挡和干扰等可见证据，主要支撑 perception/grounding；不能单独生成不可观测物理状态或因果机制真值。
- 多模态样本可以组合已对齐的视觉、传感器、布局和配置证据，但必须记录对齐方法、时间范围和各模态信息预算。

数据源可以异构，但进入同一能力任务时必须使用统一 Schema、标签语义和评分协议。允许来源缺少不适用字段，不允许用来源专属字段静默改变任务含义。

## 2. 数据分层

- L0 Source：原始只读文件、下载日志和来源快照。
- L1 Canonical：统一字段名、时间、单位、坐标和标签，但保留 source value。
- L2 Case Graph：场景、base case、intervention family、传感器、事件和因果候选边。
- L3 Benchmark：切窗后的 T1/T2/T3 样本、gold、trace 和评分元数据。
- L4 Release：公开输入、开发标签、私有测试标签、评分器和发布 manifest 分离。

每层记录 `parent_manifest_sha256`、`transform_name/version`、配置哈希、代码提交和生成时间。

## 3. ID 与匿名化

### 3.1 稳定私有 ID

- `source_id`：D01-D11。
- `source_case_key`：上游原始 case 唯一键，仅私有。
- `case_uid`：`sha256(source_id + source_version + source_case_key + private_salt)` 的截断编码。
- `family_uid`：同一 base scenario 的干预 family。
- `sample_id`：`FWB-{benchmark_version}-{task}-{split}-{opaque_id}`。

私有 salt 和反向映射不得进入公开仓库、公开 benchmark 输入或模型上下文。

### 3.2 分组键

至少维护：`source_case`、`simulation_family`、`physical_scenario`、`video_sequence`、`near_duplicate_group`、`question_template_family`。split 规则按最严格相关组执行。

## 4. Canonical case

每个 case 至少描述：

- 几何：域类型、尺寸、坐标系、区域/隧道分段。
- 火源：候选位置、燃料/反应（若已知）、HRR 或代理、点火/增长时间。
- 通风/排烟：方向、速度/流量、风机/风阀/排烟口状态。
- 传感器：稳定 ID、类型、坐标、单位、采样率、质量状态。
- 观测：时间序列、图像/视频/场图引用及可见性。
- 干预：变量、旧值、新值、实施时刻和其他保持不变的条件。
- 事件：阈值跨越、峰值、响应顺序、backlayering 等带来源的派生事件。

未知值使用明确 `null` + `missing_reason`，不得用 0、空串或默认物理值代替。

## 5. Benchmark sample 契约

机器定义见 `../schemas/benchmark_sample.schema.json`。语义结构为：

```json
{
  "sample_id": "...",
  "benchmark_version": "...",
  "task": "T1-A",
  "split": "test_ood",
  "scenario": {},
  "observations": [],
  "question": {},
  "answer": {},
  "physical_trace": {},
  "scoring_metadata": {},
  "provenance": {}
}
```

`source_id` 只记录 provenance，不决定 `task`。同一来源可以支撑多个任务，同一任务也可以由多个来源支撑；映射必须在 `06_能力-数据源-任务矩阵.md` 和版本化构建配置中显式声明。

### 5.1 `observations`

每项观测包含 `observation_id`、`modality`、`time_range_s`、`content/ref`、可见字段、单位、质量标记。公开样本只能引用发布包内资源或确定性生成物。

### 5.2 `answer`

包含任务特定的类别/排序/趋势/风险区间，以及统一的 `dominant_mechanism`、`evidence`、`uncertainty`、`missing_information`。预测 Schema 与 gold 分离，防止私有评分字段被模型复制。

### 5.3 `physical_trace`

- `initial_state`
- `mechanism_chain`
- `transitions`
- `outcome`
- `evidence_links`
- `constraints` / `known_violations`

每个节点带 `origin`：`direct_measurement`、`simulator_truth`、`deterministic_rule`、`literature`、`expert_annotation`。

### 5.4 `scoring_metadata`

保存自动计分所需的事件、容差、允许答案集、因果边和难度标签。测试集该字段私有。

### 5.5 `provenance`

至少包含 source/version、私有 case UID、父 manifest、builder/version、配置哈希、标注版本和审核状态。

## 6. 来源适配重点

### 6.1 Immersed

- 解析 `Time`、`T-*`、`C-*` 和末尾通风/流量字段。
- 文件名中的火源/排烟语义只可在私有 adapter 中使用，公开 ID 必须匿名化。
- 在字段字典确认前，`C-*` 不直接命名为 soot/CO；必须以来源说明或数值/代码证据确认。
- 同一火源不同排烟设置可形成 family，但“只改变一项”必须用元数据验证。

### 6.2 PolyUFire

- `.xls/.xlsx` 的多行表头、记录时间、测点编号和单位单独规范。
- 整理表中的机理结论与 sidewall 原始温度时序分开，不把文献汇总值伪装为本次实验传感器真值。
- 主要作为外部验证和机制知识，不单独承担完整 T3 ground truth。

### 6.3 FDS-exp 与 FD-Gen

- FDS-exp 保留 HRR、热流、热电偶、速度/压差的原始单位与实验说明。
- FD-Gen 保存生成参数、随机种子、FDS 输入、软件版本、执行日志和输出；失败 case 同样进入运行清单。
- paper-ready counterfactual family 必须明确唯一干预轴，如风速、风阀、火源位置、HRR 或障碍物。

### 6.4 视觉源

- 统一 bbox 到规范坐标时保留原格式与类别映射。
- 空标签需依据来源规范区分合法负样本与缺失/损坏标注。
- 图像重复、近重复、视频帧邻近和跨数据集转载必须扫描。
- 视觉源只提供外观、定位、干扰或 grounding 证据，不生成未观测的 CFD 物理状态。

## 7. 派生标签

### 7.1 原则

- 规则先版本化，再运行；阈值来自来源定义、物理规范、训练集统计或专家决定，并记录 origin。
- 使用训练集拟合的阈值不得查看 dev/test 分布。
- 所有事件同时保存连续数值和离散标签，避免丢失信息。

### 7.2 候选事件

- 响应起始：相对稳定基线的持续偏移。
- 响应顺序：不同传感器达到定义事件的时间次序。
- 峰值/峰时、上升率、持续超限、传播方向。
- 风险区间和机制标签。

具体阈值均为 `TBD_BY_PILOT`，P3 前由训练/校准集和专家规则冻结。

## 8. Split 设计

1. 首先按上游 case/family 去重并形成 group。
2. 再在 group 层分 train/dev/test；test 一次冻结。
3. 最后在各 split 内切窗、增强、派生问题和负样本。

计划 split：

- `train_id`、`dev_id`、`test_id`
- `test_ood_source_location`
- `test_ood_ventilation`
- `test_ood_hrr`
- `test_ood_geometry_or_layout`
- `test_external_experiment`

数量比例在 P1 盘点后通过优化约束确定，不提前固定 8:1:1。所有 OOD 轴都需有足够 group 和避免混杂的证据。

## 9. Counterfactual pair

每个 pair 必须满足：

- 相同 base case/family、初始条件和观测截止点；
- 仅 `intervention.variable` 发生声明变化；
- 其他参数的数值或哈希相等；
- 输出比较标签至少包含方向、响应顺序、事件变化和可选数值差；
- 自动 validator 能给出 mismatch 清单。

Immersed 配置文件若同时改变多个条件，只能标为“配置比较”，不得标为单变量因果反事实。

## 10. 质量门禁

- 文件：可读、哈希稳定、无意外零字节/截断/重复。
- Schema：类型、必填、枚举、单位和引用合法。
- 数值：时间单调、范围合理、采样间隔可解释、NaN/Inf 有策略。
- 标签：类别映射、空标注、bbox 范围、事件可重算。
- split：group 交集为零，近重复和模板泄漏报告通过。
- 专家：校准集、一致性、裁决、版本和冲突率记录。
- 发布：公开输入无法由名称或元数据直接推断答案。

## 11. 数据版本与发布

- 原始数据是否可再分发逐源决定；不允许时发布下载/校验脚本、哈希和派生规则。
- 每个 benchmark release 固定 `manifest.json`、Schema、数据卡、split 清单、构建器版本和 checksum。
- 许可证或来源撤回时创建新版本并登记影响，不静默删除旧结果。
