# 06 能力-数据源-任务矩阵

状态：`ACCEPTED-DESIGN-ADDENDUM`  
日期：`2026-07-14`  
适用范围：FireWorldBench v1（仅火灾领域）

## 1. 统一问题定义

FireWorldBench v1 评测 LLM、multimodal model 和 agent 能否从不完整、带噪声、多模态的火灾观测中：

1. 识别早期风险并选择需要补充的信息；
2. 恢复当前火灾状态并解释主导物理机制；
3. 预测未来演化，并对受控干预给出物理一致的反事实判断。

T1/T2/T3 是统一能力 ontology。数据集只提供观测和证据，不独立定义能力。

## 2. 能力轴

| 能力族 | 核心能力 | 必须区别于 |
|---|---|---|
| T1 预警 | 弱信号判断、异常归因、主动信息选择 | 单阈值报警、只看有无火焰 |
| T2 状态与机制 | 状态恢复、机制识别、解释一致性 | 纯 detection、场景标签复述 |
| T3 演化与反事实 | 趋势、单变量干预、状态转移链 | 序列外推、相关性比较、伪反事实 |

## 3. 数据角色轴

| 数据角色 | 可提供的证据 | 主要限制 |
|---|---|---|
| 仿真/CFD | 完整状态、空间场、参数、时间演化、可控干预 | simulation-to-reality gap；版本和参数必须冻结 |
| 受控实验 | 真实传感器噪声、缺失、物理响应和外部验证 | 状态覆盖不完整；通常缺少严格反事实 |
| 传感器时序 | 弱信号、响应顺序、趋势和故障模式 | 不一定能唯一识别隐藏状态或机制 |
| 视觉 | 可见火焰/烟气、位置、遮挡、干扰和 grounding | 不提供不可观测 CFD 真值或完整因果链 |
| 多模态组合 | 视觉、时序、布局、配置之间的证据整合 | 必须严格时间/空间对齐并控制信息预算 |

## 5. 正式样本准入规则

一个来源—任务组合只有同时满足以下条件才能进入正式 benchmark：

1. 数据许可、版本、文件哈希和用途资格通过；
2. 观测字段、单位、时间/空间坐标和缺失语义可核验；
3. case/family/sequence 和近重复分组键可执行；
4. gold origin 与任务声称匹配；
5. 能映射到统一 sample/prediction Schema 和评分指标；
6. 不依赖文件名、来源身份或模板泄漏答案；
7. 仿真、实验和视觉结果按来源角色分别报告。

不满足条件时，该组合应为 `BLOCKED`、`AUX_ONLY` 或 `N/A`，不得为了矩阵完整而制造任务。

## 6. 多来源组合规则

- 同一任务可以由多个来源构成，但先在 case 层聚合，再按预注册方式宏平均。
- 同一来源可以支撑多个任务，但必须分别生成任务特定 question、gold、evidence 和 scoring metadata。
- 仿真用于机制/反事实真值时，实验用于外部验证，二者不得被描述为同一种 ground truth。
- 视觉准确率、grounding 准确率和物理 reasoning 指标必须分开报告。
- 简单 QA 只可作为交互形式、smoke test 或校准任务；正式论文结果必须落到 T1/T2/T3 能力与证据评分。

## 7. Benchmark 与方法的阶段边界

P1-P4 先完成数据角色、任务、benchmark 和 baseline。方法方向由冻结 train/dev 的失败模式决定：

- 证据/机制链错误：候选 physical trace prompting；
- 长时状态不一致：候选结构化状态记忆；
- 机理知识缺失：候选 retrieval；
- 反事实/演化约束错误：候选计算或仿真工具调用；
- 多模态证据遗漏：候选 multimodal grounding。

这些方法在 baseline 结果前均为候选，不是既定贡献。不得通过 test 表现选择方法。

## 8. 版本与冻结

本文件是对既有框架的已批准设计附录。旧 manifest/hash、Schema、pipeline、builder、scorer 和 harness 保持可复用，不要求项目回退或重新执行完整 P1/P2 阶段。

新增真实数据时执行增量 source onboarding：只为新增或变化的来源生成版本化许可/字段/质量/角色 manifest，并重新验证受影响的 source-task 组合。新增模型时执行 model onboarding：登记模型版本、输入模态、prompt/config、运行时和预算，然后接入既有 harness。

只有能力 ontology、公共 Schema、split 原则、主指标或测试边界发生变化时，才触发对应冻结物的新版本；普通数据和模型接入不触发全项目重做。

执行顺序为：staging integration -> real benchmark build -> model onboarding -> train/dev pilot -> frozen main run -> statistics/paper export。
